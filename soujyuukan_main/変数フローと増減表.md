# soujyuukan_main.ino 変数フロー & 増減表レポート

このドキュメントは **可変抵抗 → KRS コマンド (3500〜11500)** までの変数の流れを整理し、
途中で登場する変換関数、特に `elePoly` / `rudPoly` の増減（単調性）を数値的に可視化するものです。

> 読み方: パイプラインの各段で「値が何を表しているか」「単位は何か」「次の段に渡すとき何が起きるか」を追っていけば、
> 途中の変数名に振り回されずに挙動を把握できます。

## 3. `detzoneMapping` の振る舞い (5区間折れ線)

`detzoneMapping(ary, x, *y, min, med, max)` は 4 つの閾値 `ary[0..3]` で ADC 入力を 5 区間に分割:

| 区間 | 出力 `*y` | 戻り値 |
|---|---|---|
| `x ≤ ary[0]` | `min` | 0 |
| `ary[0] < x ≤ ary[1]` | `fmap(x, ary[0], ary[1], min, med)` 線形補間 | 0 |
| `ary[1] < x ≤ ary[2]` | `med` (**不感帯・中央固定**) | 1 |
| `ary[2] < x ≤ ary[3]` | `fmap(x, ary[2], ary[3], med, max)` 線形補間 | 0 |
| `ary[3] < x` | `max` | 0 |

戻り値が 1 のとき `is_center` が立ち、PID の積分/出力更新がこのときだけ行われる (ウィンドアップ防止)。

### 注意: エレベータだけ「min/max を入れ替えて渡している」

```cpp
// 309行
is_center = detzoneMapping(elergs, fELE, &elevetor,
                           ElevatorDegMax, ElevatorDegMed, ElevatorDegMin);
// 310行
detzoneMapping(rudrgs, fRUD, &rudder,
               RudderDegMin, (RudderDegMin+RudderDegMax)/2, RudderDegMax);
```

- **エレベータ**: ADC 低 (`fELE ≤ 1120`) → `elevetor = +5°` (上げ舵)、ADC 高 → `-5°` (下げ舵)
- **ラダー**: ADC 低 (`fRUD ≤ 1550`) → `rudder = -9.3°`、ADC 高 → `+9.8°`

つまり **エレベータ軸は可変抵抗の物理的配置に合わせて意図的に反転**されています。
ここを見落とすと「サインが逆に見える」錯覚を起こします。

もう一つ: ラダーの「中央値」は `(-9.3 + 9.8)/2 = 0.25°` であり、機械的な左右非対称分だけ
正側にずれています。「中央 = 0°」ではないので、完全にオフセットを取りたければ `Trimrudder` で吸収します。

## 4. 多項式関数の増減表

ここが本題。`elePoly` と `rudPoly` は 5 次多項式で、係数を眺めてもプラス/マイナスの向きすら分かりません。
定義域全体で実数値的に評価し、単調性を確認します。

> 生成経緯: `dakaku_mapping.py` で「サーボ指令値 ↔ 実舵角 (IMU ピッチ)」の CSV を収集し、
> `numpy.polyfit(deg, krs, 5)` の結果を直接貼り付けたものと推測される
> (係数の桁数が 12 桁もあるのが手掛かり)。

### 4.1 `elePoly(x)` = 舵角[°] → KRS

```
elePoly(x) =  0.012577102329·x^5
            + 0.0194099102747·x^4
            - 0.634353764183·x^3
            - 0.815828278465·x^2
            + 185.794437174·x
            + 5701.22753031
```

定義域 (実際に PID を含めても届く範囲): **x ∈ [−5, 5]**
`ele2krs` はこれを `constrain(…, 3500, 8130)` に通します。

#### 増減表 (x ∈ [−5, 5])

| x [°] | `elePoly(x)` [KRS] | `elePoly'(x)` | 傾向 |
|---:|---:|---:|---|
| −5 | **4803.98** | +175.98 | ↗ |
| −4 | 4977.69 | +173.00 | ↗ |
| −3 | 5152.15 | +176.56 | ↗ |
| −2 | 5331.36 | +181.83 | ↗ |
| −1 | 5515.26 | +185.51 | ↗ |
|  0 | **5701.23** (中立 KRS) | +185.79 | ↗ |
| +1 | 5885.60 | +182.40 | ↗ |
| +2 | 6065.19 | +176.55 | ↗ |
| +3 | 6238.77 | +170.96 | ↗ |
| +4 | 6408.60 | +169.89 | ↗ |
| +5 | **6581.94** | +179.07 | ↗ |

- **単調増加**: `elePoly'(x)` は x ∈ [−30, 30] の広範囲スキャンで符号変化なし (=1800 点刻みで検証済)。
  定義域 [−5, 5] の中で極値なし、多価にならない。
- **感度**: 1° あたり約 **170〜186 KRS**/度。ほぼ一定ゲインの線形変換 + 微小な非線形補正。
- **中立点**: `elePoly(0) = 5701.23` (サーボ中央 7500 から約 −1800 ずれた位置が実際の水平尾翼中立)
- **クランプへの到達**: 定義域内では `4803 ≤ y ≤ 6582` で、`constrain(…, 3500, 8130)` の
  **どちらの端にも触れない**。クランプは PID 暴走や可変抵抗外れなど異常系のみを拾う安全網。

#### 上限が 8130 である意図

`rud2krs` は 3500〜11500 まで許可するが、`ele2krs` は上限を **8130** と 3400 強も狭めている。
これは機械的にエレベータが上げ舵方向に干渉するのを避けるためのハード上限と推測される
(ホーン/リンク長から決まる機械的余裕)。

### 4.2 `rudPoly(x)` = 舵角[°] → KRS

```
rudPoly(x) = −0.0105114837351·x^5
            − 0.0659075647903·x^4
            + 0.241297817826·x^3
            + 2.97624922026·x^2
            − 179.910899851·x
            + 6563.03984539
```

定義域: **x ∈ [−9.3, 9.8]** (`RudderDegMin`, `RudderDegMax`)
`rud2krs` はこれを `constrain(…, 3500, 11500)` に通します。

#### 増減表 (x ∈ [−9.3, 9.8])

| x [°] | `rudPoly(x)` [KRS] | `rudPoly'(x)` | 傾向 |
|---:|---:|---:|---|
| **−9.3** | **8537.79** | −353.76 | ↘ |
|   −9.0 | 8435.68 | −327.49 | ↘ |
|   −7.0 | 7903.91 | −221.87 | ↘ |
|   −5.0 | 7498.50 | −191.47 | ↘ |
|   −3.0 | 7120.26 | −188.39 | ↘ |
|   −1.0 | 6745.63 | −184.93 | ↘ |
|    0.0 | **6563.04** (中立 KRS) | −179.91 | ↘ |
|   +1.0 | 6386.27 | −173.55 | ↘ |
|   +3.0 | 6048.72 | −166.91 | ↘ |
|   +5.0 | 5694.01 | −197.85 | ↘ |
|   +7.0 | 5197.35 | −319.39 | ↘ |
|   +9.0 | 4307.71 | −604.72 | ↘ |
|  **+9.8** | **3754.79** | −784.95 | ↘ |

- **単調減少**: `rudPoly'(x)` は x ∈ [−30, 30] で符号変化なし。**定義域内に極値なし**。
- **符号が elevator と逆**: 正の舵角で KRS が **減る**。これは rudder 側の機械レイアウト
  (サーボホーンの回転方向と舵面の取り付け向き) でそうなっているため、
  `deg → KRS` の変換自体が負の傾きを持つように多項式がフィッティングされている。
  elePoly (↗) と rudPoly (↘) でサインが逆なのは **バグではなく仕様**。
- **感度が両端で跳ね上がる**: 中央付近 −167〜−198 KRS/度だが、x が ±9 に近づくと
  −600 を超える (折り返し地点に近いので敏感)。機械の End-Stop 近傍は
  「わずかな角度変化で KRS が大きく動く」ことに注意。
- **クランプへの到達**: 定義域の端 x = −9.3 で 8537.79、x = +9.8 で 3754.79。
  **どちらもクランプ 3500〜11500 に触れない**。elePoly 同様、
  `constrain` は可変抵抗が物理上限より広く読めた時や PID 暴走の安全網として機能。

### 4.3 elePoly vs rudPoly: なぜ符号が逆でも辻褄が合うのか

ここが混乱の発生源。以下の表を一度理解すれば、以後は悩まずに済みます。

| 方向 | ADC 入力 | `elevetor`/`rudder` | トリム後 deg | Poly → KRS | 物理的な結果 |
|---|---|---|---|---|---|
| エレベータ「上げ」 | ADC ↓ (`fELE ≤ 1120`) | +5° | +5° | 6582 (大) | 舵面が上げ舵へ |
| エレベータ「下げ」 | ADC ↑ (`fELE ≥ 2460`) | −5° | −5° | 4804 (小) | 舵面が下げ舵へ |
| ラダー「右」 | ADC ↑ (`fRUD ≥ 3100`) | +9.8° | +9.8° | 3755 (**小**) | 舵面が右へ |
| ラダー「左」 | ADC ↓ (`fRUD ≤ 1550`) | −9.3° | −9.3° | 8538 (**大**) | 舵面が左へ |

- エレベータ: 可変抵抗の物理向きを `detzoneMapping` の引数順序で反転済み。`elePoly` は普通の ↗。
- ラダー: 可変抵抗の物理向きは正順 (ADC ↑ → deg ↑)。しかし **サーボの回転向き**と舵面の
  付き方の関係で、`deg ↑ → KRS ↓` にしないと舵が逆に動いてしまう。
  そこで `rudPoly` を ↘ にフィッティングして吸収している。

つまり「反転処理」が elevator では `detzoneMapping` の引数順で、
rudder では `rudPoly` の符号で行われている、という違いがあります。

現在この 2 つは **コンパイル不能 or 動作未定義** の状態です。呼び出し元は:

```cpp

txData.E_angle = (getpos1 != -1) ? krs2ele((float)getpos1) : 0.f;
txData.R_angle = (getpos0 != -1) ? krs2rud((float)getpos0) : 0.f;
```

すなわち **BLE 側に送信する「実舵角表示」用の逆関数**。
`elePoly` / `rudPoly` の逆写像が必要ですが、以下いずれかで対処するのが筋です。

1. **直接逆写像の多項式フィット** (現状の空ボディは、この多項式係数を貼る予定の箱)。
   `dakaku_mapping.py` の CSV を `x = pitch_deg, y = servo_command` で
   `polyfit(y, x, 5)` し、逆方向の 5 次係数を求める。
2. **数値逆解法** (Newton/二分法で `elePoly(θ) = getpos1` を解く)。単調性が保証されているので
   二分法で O(log ε) で収束する。
3. **近似**: 中央付近で線形化し `θ ≈ (krs − 5701.23) / 185.79` (elevator の場合)。
   BLE 表示用なら十分な精度。

いずれにしても、現状のまま書き込むとビルドが通らないので **次に触るときの最優先修正対象**。

## 7. メインループ (30 Hz) の実行順序

1. `Potentiometer()` → `fELE/fRUD` → `elevetor/rudder` (dead zone 適用済)
2. `trimElevetor()`, `trimR udder()` → `Trimelevetor/Trimrudder` 更新
3. `degE = elevetor + Trimelevetor`, `degR = rudder + Trimrudder`
4. `errorE = 0 - currentPitch` (BLE で受信した IMU ピッチに対する偏差)
5. `krsE = ele2krs(degE)`, `krsR = rud2krs(degR)` (まず手動値で確定)
6. `is_pid && is_center` なら `tempDegE = pidCompute(…)` で **`krsE` を上書き**
   (中立外は `pidReset` してウィンドアップ防止、ただし `is_pid` モード表示 LED は維持)
7. `krs.setPos()` → サーボ指令
8. `krs.getPos()` / `krs.getTmp()` でテレメトリ取得 (温度は 1 秒/回)
9. BLE へ `ControlData` を送信
10. `vTaskDelayUntil` で 30 Hz 固定

## 8. PID 制御パスの変数フローと増減表

### 8.1 PID パイプライン

```
    meinkiban3 (BLE受信)
         │
         ├─ currentPitch [°]         機体ピッチ角 (機首上げ = 正)
         └─ currentPitchRate [°/s]   ピッチ角速度 (機首上げ方向 = 正)
         │
         ▼
    errorE = 0.0 - currentPitch      目標(0°) との誤差         427行
         │
         ▼
    pidCompute(&pidElevator, errorE, currentPitchRate)          438行
         │
         ├─ P項: kp * errorE
         ├─ I項: ki * integral        (integral += errorE * dt, ±30° クランプ)
         └─ D項: kd * (-currentPitchRate)
         │
         ▼
    tempDegE [°]                     PID 出力                   438行
         │
         ▼ (is_pid && is_center のとき)
    krsE = ele2krs(tempDegE + Trimelevetor)                     443行
         │
         ▼
    サーボへ
```

### 8.2 増減表: 機首上げ (ピッチアップ) したとき

実測により、IMU を機首上げに傾けると `currentPitch` は**負**になることが確認されている。
以下の表はこの符号規約に基づいて、機首上げ時の各変数の増減方向を追う。

| # | 変数 | 式 | 増減 | 解説 |
|---|---|---|---|---|
| 1 | `currentPitch` | BLE受信 | ↓ 負方向 | IMU取付向きにより機首上げ = 負 |
| 2 | `currentPitchRate` | BLE受信 | ↓ 負方向 | 上記と同じ符号系 (機首上げ動作中は負) |
| 3 | `errorE` | `0.0 - currentPitch` | ↑ 正方向 | pitch が負 → error は正 |
| 4 | P項 | `kp(-1.0) × errorE(正)` | ↓ 負方向 | 機首下げ舵を指令 (復元方向) |
| 5 | I項 | `ki(-0.1) × integral` | ↓ 負方向 | integral は正に蓄積 → (-0.1)×(正) = 負。P項と同方向 |
| 6 | D項 | `kd(-0.05) × (-pitchRate)` | ↓ 負方向 | pitchRate 負 → (-負) = 正 → (-0.05)×(正) = 負。ダンピング |
| 7 | `tempDegE` | P + I + D | ↓ **負** | 全項が負 → PID出力は負 (機首下げ舵) |

> P項が支配的。I項は時間経過で蓄積、D項は角速度が大きいときだけ寄与。

### 8.3 増減表: 機首下げ (ピッチダウン) したとき

| # | 変数 | 式 | 増減 | 解説 |
|---|---|---|---|---|
| 1 | `currentPitch` | BLE受信 | ↑ 正方向 | IMU取付向きにより機首下げ = 正 |
| 2 | `currentPitchRate` | BLE受信 | ↑ 正方向 | 機首下げ動作中は正 |
| 3 | `errorE` | `0.0 - currentPitch` | ↓ 負方向 | pitch が正 → error は負 |
| 4 | P項 | `kp(-1.0) × errorE(負)` | ↑ 正方向 | 機首上げ舵を指令 (復元方向) |
| 5 | I項 | `ki(-0.1) × integral` | ↑ 正方向 | integral は負に蓄積 → (-0.1)×(負) = 正 |
| 6 | D項 | `kd(-0.05) × (-pitchRate)` | ↑ 正方向 | pitchRate 正 → (-正) = 負 → (-0.05)×(負) = 正 |
| 7 | `tempDegE` | P + I + D | ↑ **正** | 全項が正 → PID出力は正 (機首上げ舵) |

### 8.4 符号規約まとめ

| 物理動作 | `currentPitch` | `errorE` | `tempDegE` | サーボの動き |
|---|---|---|---|---|
| 機首上げ | 負 | 正 | **負** (機首下げ舵) | 水平に戻す |
| 水平 | 0 | 0 | 0 | 補正なし |
| 機首下げ | 正 | 負 | **正** (機首上げ舵) | 水平に戻す |

> **注意**: `getPitch()` の符号が航空工学の慣例 (機首上げ=正) と**逆**になっている。
> これは IMU の物理的な取付向きに依存する。`Kp = -1.0` (負) にしているのは
> この符号反転を補償するためであり、制御としては正しく動作する。
> 将来の混乱を避けるなら `currentPitch` の符号を入口で反転して慣例に合わせることを検討すべき。

### 8.5 PID 出力が krsE に反映される条件分岐

```
tempDegE は常に計算される (438行)

if (is_pid) {                                    // PIDモード ON
  LED = HIGH
  if (is_center) {                               // 操縦桿が中立帯
    krsE = ele2krs(tempDegE + Trimelevetor)      // ← PID出力でサーボ駆動
  } else {
    pidReset()                                    // 操縦桿動かした → 積分リセット
  }
} else {                                         // PIDモード OFF
  LED = LOW
  pidReset()
  krsE = ele2krs(degE)                           // ← 手動舵角でサーボ駆動
}
```

`tempDegE` は PID OFF 時もログに表示される (438行で常に計算) ため、
ログの `pid:` は「もし PID が効いていたらこの値になる」のシャドウ値として使える。
