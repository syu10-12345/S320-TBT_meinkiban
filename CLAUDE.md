# S320-TBT メイン基板 / 操縦桿基板

人力飛行機 S320-TBT のフライトコンピュータ (meinkiban3) と操縦桿コントローラ (soujyuukan_main) のリポジトリ。

## リポジトリ構成

```
meinkiban3/          # メイン基板 (ESP32-S3 WROOM) — 機体側フライトコンピュータ
  meinkiban3.ino       主制御ループ。センサー読み取り・BLE通信・テレメトリ送信
  TFTand9axis_sensor.h ICM20948(9軸IMU) + TFTディスプレイ + キャリブレーション

soujyuukan_main/     # 操縦桿基板 (ESP32-S3) — パイロット側コントローラ
  soujyuukan_main.ino  ポテンショメータ入力 → フィルタ → PID → KRSサーボ制御
  PidState.h           PID状態構造体 (kp, ki, kd, integral, lastError)
  cascade_pid_guide.md カスケードPID制御の理論・実装ロードマップ
  変数フローと増減表.md  変数変換・デッドゾーン・多項式マッピングの詳細
  megrosheet.py        サーボキャリブレーション用多項式フィッティング
```

## ハードウェア・通信

- **センサー (meinkiban3 I2C0)**: ICM20948 (ピッチ/ロール), MB1242 (超音波高度計)
- **センサー (meinkiban3 I2C1)**: SDP810 (ピトー管差圧 → 対気速度)
- **GPS**: UART1 @ 921600 baud
- **フォトセンサー x2**: 前後プロペラRPM (GPIO割り込み)
- **サーボ**: KRS (近藤) エレベーター/ラダー、UART @ 115200 baud
- **BLE**: meinkiban3 (Central) ↔ soujyuukan_main "C3-CONTROL" (Peripheral)
  - meinkiban3 → soujyuukan: pitch角 (~50Hz)
  - soujyuukan → meinkiban3: 舵角・サーボ温度 (30Hz)
  - meinkiban3 → "C3-LOGGER": フルテレメトリ (~50Hz, トリガスイッチON時)

## 関連プロジェクト

**`C:\code\react-physics-lab`** — React + TypeScript で構築した機体ピッチ制御シミュレータ。
主翼・尾翼の空力パラメータ、慣性モーメント、対気速度を設定し、PIDゲインをリアルタイムに
チューニングできる。アクチュエータの時定数・通信遅延・突風外乱もモデル化しており、

## PID制御 段階的導入計画

テストフライト (TF) は全8回。TF1 は浮上しないため、機体特性の記録チャンスは **TF2〜TF8 の7回**。
限られた機会を最大限活用するため、PID制御は以下の4フェーズで段階的に導入する。

### Phase 0: データ収集 (PID OFF)

**目的**: パイロット操作と機体応答の関係を把握し、機体特性モデルを構築する。

- PID は完全に OFF。パイロットが手動で操縦する
- 以下のデータを BLE ロガーで記録する:
  - 操縦桿エレベーター角度 (degE)
  - 実サーボ角度 (E_angle)
  - 機体ピッチ角 (pitch)
  - 対気速度 (air_speed)
  - 高度 (Altitude)
  - タイムスタンプ (epoch_time)
- 収集データから以下を分析する:
  - 舵角 → ピッチ角の伝達特性 (遅れ時間、ゲイン)
  - 速度域ごとの応答変化 (V^2 依存性の確認)
  - パイロットの操舵パターン (操舵量・頻度の統計)
- react-physics-lab のシミュレータに実測パラメータを反映してゲインを事前探索する

### Phase 1: シャドウモード (PID 計算のみ・介入なし)

**目的**: PID の出力をログに残し、パイロットの判断と比較して制御則の妥当性を検証する。

- PID を内部で計算するが、サーボへの出力は **パイロット入力のみ** を使用
- ログに以下を追加記録する:
  - PID 計算値 (pidOutput)
  - P項・I項・D項 の個別値
  - 目標ピッチ角 (target = 0°)
  - 誤差 (errorE)
- パイロット入力と PID 出力の乖離を分析する:
  - PID が大きく外れていればゲイン調整が必要
  - PID がパイロットに近ければ Phase 2 に進む判断材料になる
- 実際の飛行中にパイロットに一切影響を与えないため、安全にデータを集められる

### Phase 2: 制限付きアシスト (PID クランプ範囲を段階拡大)

**目的**: PID からの舵角補正を小さい範囲から徐々に拡大し、安全に自動制御を導入する。

段階的にクランプ範囲を広げる:

| ステップ | PID出力クランプ範囲 | 備考 |
|----------|---------------------|------|
| 2-a | -1° ~ +1° | 微小補正。パイロットが違和感を感じないレベル |
| 2-b | -2° ~ +2° | 軽い安定化補助 |
| 2-c | -3° ~ +3° | パイロットの介入余地を残しつつ有効な補正 |

- 各ステップで数フライト分のデータを取り、以下を確認する:
  - ピッチの振動・発散が起きていないか
  - パイロットがオーバーライドする頻度が減っているか
  - 積分項のワインドアップが発生していないか
- 問題があれば前のステップに戻すか、ゲインを再調整する

### Phase 3: 完全制御 (-5° ~ +5°)

**目的**: エレベーターの物理的可動域全体で PID 制御を有効化する。

- PID クランプ範囲を -5° ~ +5° (エレベーター可動域全域) に拡大
- パイロットは操縦桿を中立に保ち、PID に制御を委ねる
- 操縦桿を動かした場合は手動モードに即時切替 (既存のオーバーライド機構)
- このフェーズで検討すべき追加機能:
  - 速度に応じたゲインスケジューリング (`Kp_actual = Kp_base / (V/V_ref)^2`)
  - カスケード制御 (内側ループ: ピッチレート P制御、外側ループ: ピッチ角 PI制御)
  - ピッチレート (θ̇) の BLE 送信追加

## 現在の PID 実装メモ

- PID は **soujyuukan_main** (操縦桿側) で実行。meinkiban3 から BLE で受け取った pitch を使用
- ゲイン: `Kp=-1.0, Ki=-0.1, Kd=0.0` (D項は未使用)
- 積分項クランプ: ±30° (エレベーター可動域 ±5° に対して大きい — 要調整)
- 制御周期: 30Hz (BLE遅延 20〜100ms+ が律速)
- 操縦桿がデッドゾーン外の場合、PID はリセットされ手動入力が優先される

## 既知の課題・今後の検討事項

- BLE 遅延 (20〜100ms+) が制御帯域を制限 → 高帯域の内側ループを実装する場合は meinkiban3 側で実行する必要あり
- 対気速度による制御有効性変化 (V^2 依存) への対策が未実装
- ピッチレート (θ̇) が BLE で送信されていない → カスケード制御に必要
- ラダーの PID 制御はプレースホルダーのみ
