# S320-TBT メイン基板 / 操縦桿基板

人力飛行機 S320-TBT のフライトコンピュータ (meinkiban3) と操縦桿コントローラ (soujyuukan_main) のリポジトリ。

## リポジトリ構成

```
meinkiban3/          # メイン基板 (ESP32-S3 WROOM) — 機体側フライトコンピュータ
  meinkiban3.ino       主制御ループ。センサー読み取り・ESP-NOW通信・テレメトリ送信
  TFTand9axis_sensor.h ICM20948(9軸IMU) + TFTディスプレイ + キャリブレーション

soujyuukan_main/     # 操縦桿基板 (ESP32-S3) — パイロット側コントローラ
  soujyuukan_main.ino  ポテンショメータ入力 → フィルタ → PID → KRSサーボ制御
  PidState.h           PID状態構造体 (kp, ki, kd, integral, lastError)
  cascade_pid_guide.md カスケードPID制御の理論・実装ロードマップ
  変数フローと増減表.md  変数変換・デッドゾーン・多項式マッピングの詳細
  megrosheet.py        サーボキャリブレーション用多項式フィッティング
```

## ハードウェア・通信

- **センサー (meinkiban3 I2C0)**: ICM20948 (ピッチ/ロール), MB1242 (超音波高度計)
- **センサー (meinkiban3 I2C1)**: SDP810 (ピトー管差圧 → 対気速度)
- **GPS**: UART1 @ 921600 baud
- **フォトセンサー x2**: 前後プロペラRPM (GPIO割り込み)
- **サーボ**: KRS (近藤) エレベーター/ラダー、UART @ 115200 baud
- **機体内通信**: ESP-NOW (Wi-Fi ブロードキャスト, channel 1)
  - 宛先は `FF:FF:FF:FF:FF:FF` に統一し、構造体内の `magic=0x53333230` と `role` フィールドで受信側がフィルタ
  - role 定数: `ROLE_MEINKIBAN3=1`, `ROLE_SOUJYUUKAN=2`, `ROLE_LOGGER=3`
  - meinkiban3 → soujyuukan: `NavigationData` (pitch, pitch_rate) (~50Hz)
  - soujyuukan → meinkiban3: `ControlData` (舵角・サーボ温度・is_assisted) (30Hz)
  - meinkiban3 → LOGGER: `FullTelemetryPacket` (~50Hz, トリガスイッチON時)
  - 通信断検知: soujyuukan 側 `PITCH_LINK_TIMEOUT_MS=300ms` で PID 停止、meinkiban3 側 `g_lastRecvFromSoujyuukanMs` で検知

## PID制御 段階的導入計画

テストフライト (TF) は全8回。TF1 は浮上しないため、機体特性の記録チャンスは **TF2〜TF8 の7回**。
限られた機会を最大限活用するため、PID制御は以下の4フェーズで段階的に導入する。

### Phase 0: データ収集 (PID OFF)

**目的**: パイロット操作と機体応答の関係を把握し、機体特性モデルを構築する。

- PID は完全に OFF。パイロットが手動で操縦する
- 以下のデータを ESP-NOW ロガーで記録する:
  - 操縦桿エレベーター角度 (degE)
  - 実サーボ角度 (E_angle)
  - 機体ピッチ角 (pitch)
  - 対気速度 (air_speed)
  - 高度 (Altitude)
  - タイムスタンプ (epoch_time)
- 収集データから以下を分析する:
  - 舵角 → ピッチ角の伝達特性 (遅れ時間、ゲイン)
  - 速度域ごとの応答変化 (V^2 依存性の確認)
  - パイロットの操舵パターン (操舵量・頻度の統計)
- react-physics-lab のシミュレータに実測パラメータを反映してゲインを事前探索する

### Phase 1: シャドウモード (PID 計算のみ・介入なし)

**目的**: PID の出力をログに残し、パイロットの判断と比較して制御則の妥当性を検証する。

- PID を内部で計算するが、サーボへの出力は **パイロット入力のみ** を使用
- ログに以下を追加記録する:
  - PID 計算値 (pidOutput)
  - P項・I項・D項 の個別値
  - 目標ピッチ角 (target = 0°)
  - 誤差 (errorE)
- パイロット入力と PID 出力の乖離を分析する:
  - PID が大きく外れていればゲイン調整が必要
  - PID がパイロットに近ければ Phase 2 に進む判断材料になる
- 実際の飛行中にパイロットに一切影響を与えないため、安全にデータを集められる

### Phase 2: 制限付きアシスト (PID クランプ範囲を段階拡大)

**目的**: PID からの舵角補正を小さい範囲から徐々に拡大し、安全に自動制御を導入する。

段階的にクランプ範囲を広げる:

| ステップ | PID出力クランプ範囲 | 備考 |
|----------|---------------------|------|
| 2-a | -1° ~ +1° | 微小補正。パイロットが違和感を感じないレベル |
| 2-b | -2° ~ +2° | 軽い安定化補助 |
| 2-c | -3° ~ +3° | パイロットの介入余地を残しつつ有効な補正 |

- 各ステップで数フライト分のデータを取り、以下を確認する:
  - ピッチの振動・発散が起きていないか
  - パイロットがオーバーライドする頻度が減っているか
  - 積分項のワインドアップが発生していないか
- 問題があれば前のステップに戻すか、ゲインを再調整する

### Phase 3: 完全制御 (-5° ~ +5°)

**目的**: エレベーターの物理的可動域全体で PID 制御を有効化する。

- PID クランプ範囲を -5° ~ +5° (エレベーター可動域全域) に拡大
- パイロットは操縦桿を中立に保ち、PID に制御を委ねる
- 操縦桿を動かした場合は手動モードに即時切替 (既存のオーバーライド機構)
- このフェーズで検討すべき追加機能:
  - 速度に応じたゲインスケジューリング (`Kp_actual = Kp_base / (V/V_ref)^2`)
  - カスケード制御 (内側ループ: ピッチレート P制御、外側ループ: ピッチ角 PI制御)
  - ピッチレート (θ̇) の ESP-NOW 送信追加

## 現在の PID 実装 (2026-04-16 時点)

- PID は **soujyuukan_main** (操縦桿側) で実行。meinkiban3 から ESP-NOW で受け取った pitch と pitch_rate を使用
- ゲイン: `Kp=-1.0, Ki=-0.1, Kd=-0.05`
- **D項はジャイロ生値 (pitch_rate) を直接使用** — 数値微分は廃止済み
- 積分項クランプ: ±30° (エレベーター可動域 ±5° に対して大きい — 要調整)
- 制御周期: 30Hz (ESP-NOW 通信遅延が律速 — 実測要再計測)
- 操縦桿がデッドゾーン外の場合、PID はリセットされ手動入力が優先される
- `tempDegE` は PID OFF 時も常に計算される (シャドウログとして機能)

### 符号規約 (実測確認済み)

IMU の取付向きにより、航空慣例と符号が逆:
- **機首上げ → `currentPitch` は負**, 機首下げ → 正
- `Kp=-1.0` の負符号がこの反転を補償している
- 詳細は `soujyuukan_main/変数フローと増減表.md` セクション 8 を参照

### NavigationData 構造体 (ESP-NOW 通信)

```cpp
struct NavigationData {
  float pitch;       // ピッチ角 [°]
  float pitch_rate;  // ジャイロ Y軸 [°/s]
};
```
meinkiban3 と soujyuukan_main の両方で同じ定義が必要 (`#pragma pack(push, 1)`)。
片方だけ更新すると `sizeof` 不一致で ESP-NOW データが無視される (`onRecv` 冒頭の `len != sizeof(...)` ガード)。

## PID 制御方式の設計判断 (2026-04-16 議論まとめ)

### 検討した2案

| | カスケード二重ループ | 単ループ + ジャイロ D項 |
|---|---|---|
| 構造 | 外側: ピッチ角→目標ピッチレート / 内側: ピッチレート→舵角 | 標準 PID だが D項にジャイロ角速度を直接使用 |
| チューニング | 4〜7パラメータ、内側→外側の順序制約あり | 3パラメータ (Kp, Ki, Kd) |
| 通信遅延の影響 | 内側ループも通信律速 (50Hz上限) → 帯域分離不可 | D項は受信時点の最新値で十分 |
| 必要な変更 | サーボ制御の移設 or 通信高速化 | ESP-NOW パケットにジャイロ値追加のみ |
| TF7回で検証可能か | 厳しい | 十分可能 |

### 採用: 単ループ + ジャイロ D項

**理由**: 機体内通信の遅延 (意思決定当時は BLE; 現在は ESP-NOW に移行済。実遅延は要再計測) が
カスケードの内側ループを殺す。内側を外側の 3〜10 倍速くする前提が成り立たない現アーキテクチャでは、
カスケードの理論的優位性を活かせない。

ジャイロ D項は数学的に `d(error)/dt = -pitch_rate` であり、近似ではなく正確。
カスケードの内側ループ (ピッチレート P制御) とほぼ等価の効果を、
実装コスト 1/5 で得られる。

### カスケードが有利になる条件 (将来の参考)

- サーボ制御を meinkiban3 に移設し、IMU→PID内側→サーボを機体間通信を介さず 100Hz+ で回せる場合
- または有線通信で遅延 1ms 以下にできた場合
- ESP-NOW 実測遅延が十分短ければ現アーキテクチャでも再検討可能 (要計測)

### 段階的導入計画の補足 (議論で得た改善点)

- **Phase 0 と Phase 1 は同時実施可能** — シャドウ計算は操縦に影響しないので TF2 初回から PID ログを取れる。これで 1〜2 フライト分の余裕が生まれる
- **Phase 2 のクランプには anti-windup が必要** — PID 出力をクランプしても積分項は蓄積し続ける。クランプに当たっている間は積分更新を停止する (back-calculation) 実装が Phase 2 開始前に必要
- **通信遅延込みのシミュレーション** — react-physics-lab で ±1° クランプ + 実測 ESP-NOW 遅延の条件で発振しないか事前検証すべき
- **V² 問題は Phase 2 から対応開始** — 少なくとも air_speed をログに含め、速度域ごとに PID 出力の妥当性を分析する
- **Go/No-Go 基準を定量化** — ピッチ振動振幅 ±2° 以内、クランプ張り付き時間 30% 以下、パイロットオーバーライド回数 N回以下 など

## 既知の課題・今後の検討事項

- 機体内通信 (ESP-NOW) の遅延が制御帯域を制限する可能性あり → 実測で十分短ければ単ループで問題ないが、高帯域の内側ループが必要と判明した場合は meinkiban3 側で実行する設計変更が必要
- 対気速度による制御有効性変化 (V^2 依存) への対策が未実装
- ラダーの PID 制御はプレースホルダーのみ
- `integralMax=30°` はエレベーター可動域 ±5° に対して大きすぎる可能性あり
- `getPitch()` の符号が航空慣例と逆 — IMU 取付向き依存。入口で反転して慣例に合わせることを検討
